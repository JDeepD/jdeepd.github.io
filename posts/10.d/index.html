June 22, 2022 3-minute read
GSoC

The branch that I am working on is derived from Hariom’s work in 2020-21. It has over 30 commits and many of the commits are yet to be reviewed by the mailing list. So we(me and my mentors) decided its time we slowly put some commits in the mailing list for reviews. Any requested change can be done easily right now than later.

Apart from that, there is one more bug that I am trying to figure out how to solve. So, generally Git provides the option to use many non-UTF-8 encoding for commit messages. See this. Now, we will be using pretty_print_ref() for printing commit messages which does not support non-UTF-8 encoding.

Now, originally git-log could display non-UTF encoded messages using the following fn:

void repo_format_commit_message(struct repository *r,
				const struct commit *commit,
				const char *format, struct strbuf *sb,
				const struct pretty_print_context *pretty_ctx)
{
	struct format_commit_context context = {
		.repository = r,
		.commit = commit,
		.pretty_ctx = pretty_ctx,
		.wrap_start = sb->len
	};
	const char *output_enc = pretty_ctx->output_encoding;
	const char *utf8 = "UTF-8";

	strbuf_expand(sb, format, format_commit_item, &context);
	rewrap_message_tail(sb, &context, 0, 0, 0);

	/*
	 * Convert output to an actual output encoding; note that
	 * format_commit_item() will always use UTF-8, so we don't
	 * have to bother if that's what the output wants.
	 */
	if (output_enc) {
		if (same_encoding(utf8, output_enc))
			output_enc = NULL;
	} else {
		if (context.commit_encoding &&
		    !same_encoding(context.commit_encoding, utf8))
			output_enc = context.commit_encoding;
	}

	if (output_enc) {
		size_t outsz;
		char *out = reencode_string_len(sb->buf, sb->len,
						output_enc, utf8, &outsz);
		if (out)
			strbuf_attach(sb, out, outsz, outsz + 1);
	}

	free(context.commit_encoding);
	repo_unuse_commit_buffer(r, commit, context.message);
}

Commit encoding can be set by

git config i18n.commitEncoding <Encoding>

First, repo_format_commit_message() will check if the requested output encoding is same as UTF-8(it is by default). If its set to something other than UTF-8, then reencode_string_len() will reecode the commit message in the given format and return the reencoded message.

This function is called by pretty_print_commit() which is currently used to print commit bodies when invoked with pretty formats.

Now, the problem is I can’t simply make this fn call from pretty_print_ref() since it does not take a struct pretty_print_context as an argument so I cant pass the encoding through that struct.

There is another fn get_log_output_encoding() which (probably) gives the encoding set by the user(but I am yet to try it). So I will probably try to do something with that if it works. Also have to make sure that it does not break other functions which uses pretty_print_ref().

So for this week, this is the end. Things are starting to get a bit more complicated but also interesting as I am learning something new about the codebase everyday. Never in the dreams I could have thought such complex things could be done in C but here we are.

Seeya next week.
